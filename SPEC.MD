# AI 小说写作系统规格

## 0. 文档定位

`SPEC.MD` 是 AI 小说写作系统的项目级规格入口，定义系统必须实现什么、哪些边界不可破坏、运行产物如何流转、章节如何进入故事正史，以及后续实现必须满足的验收口径。

`writing-flow-design.md` 负责解释为什么采用这套写作流程、状态约束和角色分工。`SPEC.MD` 不重复理念论证，只把设计理念收敛为可执行规格。

本文中的结构名称、字段意图和状态枚举用于约束产品能力与实现边界；它们不是最终代码接口、数据库表结构或 API schema。后续实现可以调整字段名和存储形式，但不能破坏本文定义的语义、边界和验收条件。

当前设计研究目录的产品级文档只保留两份：

- `SPEC.MD`：定义系统目标、硬边界、核心闭环、数据产物、状态流转和验收口径。
- `writing-flow-design.md`：解释设计理念、写作流程、取舍原因和参考项目吸收点。

## 1. 项目目标

本项目是一个面向长篇小说自动生成的 Agent Harness。

目标是：输入一句或一段创作需求后，系统能够持续生成剧情丰富、人物稳定、设定连贯、伏笔可追踪、状态可回滚且 token 成本可控的长篇小说。

系统重点解决：

- 长篇剧情漂移。
- 角色动机、口吻、关系和认知边界失真。
- 世界设定和已发生事实冲突。
- 用户要求在长任务中衰减。
- 伏笔遗忘、污染、误回收或无痕废弃。
- 单章节奏失控、章节任务缺失、连续章节同构。
- 正文生成后状态无法可靠回收。
- 长任务中断、重复生成、上下文损坏后无法恢复。

本项目不是通用 agent 测评平台，不是简单 prompt 包装器，也不是一次性章节生成器。所有 harness 设计必须服务于长篇小说持续创作本身。

## 2. 设计硬边界

### 2.1 完整规格先行

本项目采用完整规格先行、按依赖顺序施工的方式推进。

设计阶段应先明确最终系统的核心抽象、运行闭环、状态边界、数据产物、失败处理和不可破坏的不变量。实现阶段可以按依赖顺序分步施工，但每一步都必须是最终系统的一部分，不允许为了快速跑通引入后续需要推翻的临时流程。

阶段化实现只表示施工顺序，不表示产品设计范围缩水。

### 2.2 创作宽，落盘严

系统必须遵守：

```text
正文生成阶段要宽，事实落盘阶段要严。
灵感产生阶段要宽，故事正史更新阶段要严。
章节内给正文写作者自由，章节间用状态系统收束。
```

正文写作者可以自由设计场景、对白、动作、心理、局部冲突、微反转和章节钩子，但不能静默改变故事正史、用户硬要求、角色不变量、世界核心规则、核心伏笔和高风险长期方向。

### 2.3 默认自动化

默认运行模式为自动模式：普通章节流程自动推进，只有高风险重大变更提案阻塞并等待确认。

系统内部必须保留风险等级、重大变更提案、提交记录和回滚记录，以便处理重大不可逆变化。

自动模式不表示所有推断都可以静默固化。以下内容即使未达到重大变更，也必须标记确认状态或进入暂停确认：

- 系统从用户短需求中推断出的长期偏好，准备升级为持续不变量时。
- 中风险关系转折会改变后续多个章节互动方式时。
- 局部规则例外会影响后续多章策略时。
- 用户要求、类型承诺或读者承诺存在多种合理解释时。
- 系统需要在两个长期方向之间做不可轻易回退的选择时。

后续可支持三档确认策略：

- 自动模式：默认全自动，仅重大变更提案阻塞。
- 故事弧复盘模式：每个故事弧复盘时允许人工确认。
- 严格模式：高风险状态补丁和重大变更提案都需要人工确认。

### 2.4 不静默改写故事正史

已经进入故事正史的内容不能被普通修订流程无痕覆盖。已提交正史的修正必须表现为显式修正、回滚事件、补丁逆转或新的章节事务。

候选正文、候选状态补丁、候选索引和待应用补丁都不是故事正史。只有 `ChapterCommit` 成功后，绑定的正文版本、状态版本、审稿结论和提交记录才进入故事正史。

### 2.5 状态变更必须有证据

状态账本不能凭空更新。任何角色状态、关系变化、伏笔状态、世界规则、已发生事件或用户要求满足情况的变化，都必须能追溯到用户输入、已提交正文、提案决议、人工确认或显式系统事件。

审稿报告只能证明“检查过程和判断结果存在”，不能单独充当事实来源。状态变化必须能回到正文证据、用户输入、提案决议或人工确认；不能用模型生成的审稿结论证明模型生成的状态变更。

缺少证据的状态补丁不能提交。

### 2.6 工程安全边界

系统必须把故事安全和工程安全分开处理。叙事可见性过滤只解决故事秘密泄露，不等于工程安全。

工程实现必须满足：

- API key、模型凭据和本地敏感配置不能进入正文、上下文包、审稿报告、状态补丁或提交记录。
- 模型调用、文件读写、外部网络访问和工具权限应分层隔离，并保留审计。
- 默认只写入当前作品工作区和系统定义的数据产物位置。
- 参考资料、开源项目源码、缓存目录和外部文件不得被故事流程静默修改。
- 用户正文、参考小说、角色台词和模型生成内容都不能越权改变工具权限、文件边界或提交规则。
- 检测到 prompt injection、越权文件写入、密钥泄露风险或外部调用越权时，应暂停相关流程并生成安全事件，而不是继续章节事务。

## 3. 核心概念

### 3.1 故事正史

故事正史是系统承认已经发生、已经承诺、后续必须继承的内容集合。

故事正史至少包括：

- 用户硬要求和持续不变量。
- 已提交章节正文版本。
- 已发生事件。
- 角色稳定层和动态层。
- 角色关系变化及证据。
- 世界硬规则、局部规则和已确认例外。
- 伏笔状态、历史动作和回收条件。
- 长篇指南针、当前故事弧、重大变更提案决议记录，以及已经写回上游产物的已确认变更。
- `ChapterCommit` 记录和状态版本历史。

### 3.2 候选产物

候选产物是章节事务中生成但尚未进入故事正史的内容。

包括：

- 候选章节任务卡。
- 正文初稿和修订稿。
- 审稿报告。
- 候选状态补丁。
- 待应用状态补丁。
- 无补丁声明。
- 冲突报告和不变量检查报告。
- 候选上下文索引。
- 重大变更提案。
- 滚动复盘报告和候选调整。
- 安全事件。

候选产物可以被修订、作废、替换或升级为提案，但不能被当成正史引用。

### 3.3 派生产物

派生产物由故事正史重建，不是新的正史来源。

包括：

- 下一章上下文包。
- 检索索引。
- 角色/伏笔/世界规则的摘要视图。
- 面向正文写作者的可见性过滤视图。
- 进度概览和压缩上下文。

派生产物损坏时，应从最近一次成功的 `ChapterCommit` 和状态账本重建。

## 4. 角色与职责

### 4.1 流程协调者

负责判断当前流程阶段、调度角色、处理路由、控制上下文预算、管理断点恢复和决定是否进入滚动复盘。

流程协调者不直接写正文，除非实现处于极简运行形态且角色被工程合并。

### 4.2 故事规划者

负责创作需求解析、读者承诺、故事核心引擎、长篇指南针、当前故事弧、章节任务卡和节奏预算。

故事规划者负责方向和结构，不替正文写作者决定所有表达细节。

### 4.3 正文写作者

负责把章节任务卡转化为可读正文，并在自由区域内创造场景、对白、冲突、细节和章节钩子。

正文写作者必须遵守硬事实、角色认知边界、用户硬要求、世界规则和任务边界。

### 4.4 审稿者

负责检查章节任务完成度、可读性、节奏、读者期待、角色一致性、设定冲突、伏笔问题和明显指令遗漏。

审稿者默认不直接改正文，只输出审稿结论、问题证据、修订目标和后续动作。

### 4.5 故事状态维护者

负责从已通过审稿的候选正文中抽取候选状态补丁，验证证据、权限、冲突和风险等级，并维护故事正史账本。

故事状态维护者不写正文、不设计未来剧情、不判断章节是否好看、不删除故事正史、不在无证据时修改账本、不静默覆盖冲突状态。

### 4.6 可选增强角色

以下角色可先作为审稿者或故事状态维护者的检查清单，后续按复杂度拆分：

- 角色维护者。
- 伏笔维护者。
- 读者模拟器。

系统不应默认形成过重的委员会式写作流程。

## 5. 总体运行闭环

系统采用两层循环。

### 5.1 长线规划循环

```text
创作需求
→ 需求解析
→ 读者承诺
→ 故事核心引擎
→ 角色-世界-冲突螺旋校验
→ 长篇指南针
→ 当前故事弧
```

长线规划负责方向，不提前锁死所有章节。

长线规划的必备产物：

- `RequirementLedger`：用户要求账本。
- `ReaderPromise`：读者承诺。
- `StoryEngine`：故事核心引擎。
- `NovelCompass`：长篇指南针。
- `ArcPlan`：当前故事弧规划。

### 5.2 单章生产循环

```text
上下文重建
→ 章节任务卡
→ 任务卡预检
→ 正文初稿
→ 正文写作者自检
→ 审稿报告
→ 必要时一次自动修订
→ 状态变更补丁
→ 补丁校验
→ 章节提交门禁
→ ChapterCommit
→ 下一章上下文重建
```

单章生产循环是章节事务，不是单向流水线。每个节点必须有明确出口：进入下一步、局部修订、回到上游、暂停提交、生成重大变更提案或从检查点恢复。

### 5.3 滚动复盘循环

滚动复盘用于处理连续偏差、故事弧结束、节奏同构、伏笔逾期、角色目标失焦、全局新要求、上下文膨胀或状态冲突。

滚动复盘输出是候选调整、下一故事弧规划或重大变更提案。它不能自动改写已提交的故事正史。

## 6. 运行状态机

### 6.1 作品阶段

作品阶段用于描述整本书所处状态：

- `Intake`：创作需求解析中。
- `Planning`：读者承诺、故事核心引擎、指南针或故事弧构建中。
- `ArcReady`：当前故事弧可进入章节事务。
- `ChapterTransaction`：单章事务进行中。
- `RollingReview`：滚动复盘中。
- `PausedForConfirmation`：等待用户确认重大变更或高风险事项。
- `SecurityPaused`：检测到工程安全边界问题，暂停相关流程。
- `Completed`：作品完成或当前目标结束。

### 6.2 章节事务状态

章节事务状态用于描述当前章节卡在哪个环节：

- `ContextRebuilding`
- `TaskCardDrafting`
- `TaskCardPrecheck`
- `Drafting`
- `WriterSelfCheck`
- `EditorialReview`
- `Revision`
- `StatePatchExtracting`
- `PatchValidating`
- `CommitGate`
- `Committed`
- `Blocked`
- `SecurityPaused`
- `Invalidated`

同一章节事务可以多次产生候选版本。重试时必须生成新的候选版本，并保留审计记录。

状态机实现必须定义允许转移、触发事件、守卫条件、责任角色和终态。不能只依赖枚举名推断流程。

最低转移约束：

- 只有上下文重建通过后，才能生成或重做任务卡。
- 只有任务卡预检通过后，才能进入正文创作。
- 只有审稿结论为 `Pass`，才能进入状态补丁提取。
- `LocalRevisionRequired` 必须进入修订并重新审稿。
- `RewriteRequired` 必须回到正文创作或更上游节点。
- `MajorChangeProposalRequired` 必须进入提案流程。
- `PatchValidating` 失败不能进入 `CommitGate`。
- 检测到工程安全事件时，必须进入 `SecurityPaused`，并生成安全事件。
- `Committed` 为成功终态；`Invalidated` 和 `Blocked` 为非成功终态，必须记录原因和恢复入口。

### 6.3 产物状态

产物状态至少包括：

- `Candidate`：候选。
- `Pending`：已通过验证，等待提交。
- `Staged`：已被提交事务暂存，但尚未成为正史可见内容。
- `Committed`：已随 `ChapterCommit` 入正史。
- `Derived`：由正史重建的派生产物。
- `Rejected`：已被明确拒绝，不再进入当前流程。
- `Conflict`：存在未解决冲突，需要冲突报告、修订或提案。
- `Superseded`：被新的候选版本替代。
- `Invalidated`：因上游变化或失败被作废。
- `Archived`：保留审计但不再参与当前流程。

任何产物状态变化都必须记录触发事件、来源版本、责任角色和后续路由。

## 7. 核心数据产物

### 7.1 RequirementLedger

记录用户明确要求、隐含偏好、禁止事项、持续不变量和需要确认的模糊点。

必须支持：

- 区分用户原话、系统推断和后续确认。
- 区分硬要求、软偏好、一次性要求和持续不变量。
- 记录作用范围、优先级、状态、最近检查章节和证据。

持续不变量不能简单标记为已满足，应在每章审稿和状态同步中持续检查。

### 7.2 ReaderPromise

定义本书向读者许诺的类型体验、情绪回路、主要看点、长期悬念、节奏倾向和风格边界。

读者承诺不是剧情细纲，而是长期校准标准。

### 7.3 StoryEngine

回答长篇能持续推进的基本问题：

- 主角想要什么。
- 主角为什么行动。
- 阻力来自哪里。
- 世界规则如何制造持续冲突。
- 长期悬念是什么。
- 故事为什么可以写成长篇。

缺少主角目标、长期阻力、可重复升级的冲突机制、阶段性奖励或更大真相时，系统不能直接进入章节写作。

### 7.4 NovelCompass

保存全书级方向感：

- 作品类型和读者承诺。
- 主角长期目标。
- 世界核心规则。
- 主要势力或阻力。
- 核心秘密或终局真相。
- 大阶段方向。
- 当前阶段目标。
- 不可破坏的长期约束。

指南针提供方向，不提前写死所有章节。

### 7.5 ArcPlan

描述当前故事弧。

应包括：

- 弧线目标。
- 起始局势。
- 主要冲突升级。
- 关键转折。
- 需要推进或回收的伏笔。
- 角色关系变化方向。
- 结束时的新局势。

后续故事弧只保留方向和候选可能，不提前锁死。

### 7.6 ChapterTaskCard

章节任务卡是故事规划者交给正文写作者的核心接口。

必须包含三类信息：

- 硬约束：用户硬要求、已发生事实、禁止透露信息、角色认知边界、世界硬规则和不可破坏的不变量。
- 目标变化：本章希望造成的可追踪变化，包括关系变化、信息变化、局势变化、情绪位置变化、读者理解变化或伏笔动作；不要求每章都制造硬事件或不可逆事件。
- 创作提示：戏剧问题、代价、反转空间、节奏预算、应避免的无聊写法和自由发挥区。

同时必须表达本章的读者期待和验收口径：本章要制造、满足、延迟或反转什么期待；审稿者应按本章意图判断是否有效，而不是套用统一强冲突模板。

任务卡不是正文梗概，不能把正文写作者压成执行清单。

任务卡必须通过预检。若目标过载、约束冲突、上下文不足、节奏预算不合理、用户要求不清或当前故事弧不足以支撑本章，不能直接进入正文创作。

### 7.7 Draft

正文稿包括初稿和修订稿。

正文稿必须绑定：

- 所属章节事务。
- 来源任务卡版本。
- 使用的上下文版本。
- 生成者。
- 生成时间或顺序。
- 自检结果。

正文稿在 `ChapterCommit` 之前始终是候选文本。

### 7.8 WriterSelfCheck

正文写作者自检只负责明显问题：

- 是否遗漏任务卡中的核心目标。
- 是否明显违反硬约束。
- 是否存在局部风格、格式、重复或连续性小问题。
- 是否能在不改变结构的前提下做局部打磨。

正文写作者自检不替代审稿者，不负责结构性审稿和故事正史校验。

### 7.9 EditorReport

审稿报告必须包含：

- 章节任务完成度。
- 一致性问题。
- 可读性和章节效果问题。
- 章节效果追踪，包括事件推进、读者理解、人物关系、情绪位置、压力结构或伏笔状态中至少一种可追踪变化。
- 已接受的创造性达成方式，以及为什么它没有破坏章节意图或硬边界。
- 角色、设定、伏笔、节奏和读者期待检查。
- 问题证据。
- 修订目标。
- 审稿结论。

审稿结论必须落入以下动作之一：

- `Pass`：可进入状态同步。
- `LocalRevisionRequired`：需要局部修订。
- `RewriteRequired`：需要回炉重写。
- `MajorChangeProposalRequired`：正文出现有价值但会改写正史的分岔。
- `PauseCommit`：信息不足、基础设施异常、用户确认缺失或上下文不可判定，暂不提交。

只有 `Pass` 可以进入 `ChapterCommit` 路径。其他结论只能路由到修订、重写、重大变更提案、暂停或上游重整。

### 7.10 StatePatch

状态补丁从已通过审稿的候选正文中抽取，描述本章造成的结构化事实变化。

补丁必须包含：

- 来源章节事务。
- 来源正文版本。
- 基准状态版本。
- 变更目标。
- 证据。
- 风险等级。
- 置信度或确定性说明。
- 字段权限检查结果。
- 冲突检查结果。
- 回滚提示。

状态补丁在 `ChapterCommit` 之前只能是候选或待应用，不能提前写入故事正史。

风险等级必须形成可执行路由：

- 低风险补丁：证据充分且不触及核心约束时，可以进入 `Pending`。
- 中风险补丁：可以进入 `Pending`，但必须标注风险、影响范围和是否需要确认；若涉及用户隐含偏好、重要关系转折、局部规则例外或长期节奏改变，应按确认策略决定是否暂停。
- 高风险补丁：不能进入普通 `Pending`，必须转为 `MajorChangeProposal`、冲突报告或等待确认。

### 7.11 NoStatePatchDeclaration

无补丁声明是正式产物，用于说明本章不产生可提交状态变化。

无补丁声明必须绑定：

- 来源章节事务。
- 来源正文版本。
- 审稿报告版本。
- 状态抽取尝试记录。
- 基准状态版本。
- 无补丁理由。
- 检查者。

无补丁声明必须通过以下检查：

- 已对通过审稿的候选正文执行状态抽取尝试。
- 已检查角色状态、关系、伏笔、世界规则、用户要求和上下文继承项。
- 已确认本章没有需要进入状态账本的事实变化，或变化只属于审稿报告中的章节效果追踪。
- 已说明为什么不需要生成状态补丁。
- 已确认不存在被跳过的中风险或高风险变化。

不能因为状态维护者没有生成补丁，就默认本章无需补丁。只有无补丁声明通过检查后，提交门禁才可以接受“无需补丁”。

### 7.12 PatchValidationReport

补丁校验报告必须检查：

- 是否有正文证据。
- 是否与基准状态版本匹配。
- 是否修改了允许修改的字段类别。
- 是否违反用户硬要求。
- 是否破坏角色不变量。
- 是否改变世界核心规则。
- 是否污染核心伏笔。
- 是否触发重大变更提案。
- 是否与现有故事正史冲突。

补丁未通过不自动否定正文文本价值，但会阻断 `ChapterCommit`。

### 7.13 ConflictReport And InvariantCheckReport

冲突报告和不变量检查报告用于记录不能静默提交的问题。

必须包含：

- 来源产物和版本。
- 冲突类型或不变量类型。
- 涉及的用户要求、角色、伏笔、世界规则或章节事务。
- 证据位置。
- 严重程度。
- 建议路由：修订、重写、补丁修正、重大变更提案、等待用户确认或作废候选产物。
- 对已生成下游候选产物的影响。

冲突报告本身不修改故事正史，只决定后续路由。

### 7.14 MajorChangeProposal

重大变更提案用于保存有价值但会改变故事正史或长期方向的分岔。

触发场景包括：

- 改变主角长期目标、核心性格或道德底线。
- 改变世界核心规则。
- 重要角色死亡、黑化、背叛或洗白。
- 提前揭示核心秘密。
- 提前回收、废弃或转化核心伏笔。
- 大幅偏离故事弧目标。
- 改变作品类型、文风方向或叙事结构。
- 引入长期主角级新角色。
- 永久改变能力系统上限。
- 把临时配角升级为主线角色。

提案被接受后，应先更新对应上游产物，例如长篇指南针、当前故事弧、任务卡或状态账本，再重新检查、修订或重写候选章节。提案本身不是故事正史，不能绕过 `ChapterCommit`。

重大变更提案不必立即压成接受或拒绝。系统可以保留备选章节分支、分岔片段、可读性判断和张力来源，供后续比较；但这些备选内容在被确认并经过上游更新和章节事务重检前，不能进入故事正史。

### 7.15 ChapterCommit

章节提交记录是章节进入故事正史的发布边界。

必须绑定：

- 章节编号或章节事务 ID。
- 任务卡版本。
- 正文版本。
- 审稿报告版本。
- 待应用状态补丁或无补丁声明。
- 基准状态版本和新状态版本。
- 上下文版本。
- 检查点。
- 审计事件。

`ChapterCommit` 成功后，正文版本、状态版本和提交记录才可以被后续章节继承。后续章节只能继承已成功 `Indexed` 的派生索引，或显式标记为临时且来源版本完整的派生视图。

### 7.16 ContextBundle

上下文包是下一章写作前的派生产物。

必须包含当前章节必要信息：

- 当前章节任务卡。
- 读者承诺。
- 当前故事弧目标。
- 最近章节摘要。
- 相关角色当前状态。
- 相关世界规则。
- 相关伏笔。
- 用户硬要求。
- 禁止揭示、角色禁止知道和硬约束。

上下文包必须经过可见性过滤，避免把角色不应知道的信息、读者暂不应看到的真相或与本章无关的远期秘密交给正文写作者。

上下文包必须绑定：

- 来源 `ChapterCommit`。
- 来源状态版本。
- 索引版本。
- 可见性策略版本。
- 生成时的 token 预算。

回滚、补丁逆转、历史事实修正、可见性策略更新或来源索引失效后，相关上下文包必须失效并重建。

### 7.17 RollingReviewReport

滚动复盘报告是滚动复盘的输出产物。

必须包含：

- 触发原因。
- 当前故事弧是否仍然有效。
- 读者承诺是否保持。
- 主角目标是否清晰。
- 主要角色关系是否有推进或失焦。
- 伏笔是否过多、遗忘、逾期或被污染。
- 章节节奏是否重复。
- 连续偏差根因。
- 候选调整。
- 需要更新的上游产物。
- 是否需要重大变更提案。

滚动复盘报告本身不是故事正史。只有后续章节事务、提案决议或显式规划更新提交后，复盘结论才会影响正史。

### 7.18 SecurityEvent

安全事件用于记录工程安全边界被触发的情况。

必须包含：

- 触发来源。
- 涉及的工具、文件、外部调用或敏感配置。
- 风险类型。
- 被阻断的动作。
- 影响范围。
- 恢复条件。
- 是否需要用户确认。

安全事件本身不进入故事正史，但会阻断相关章节事务，直到恢复条件满足。

## 8. 状态账本

状态账本是自定义小说运行账本，围绕章节事务和长篇连续性建立。

必须采用追加式审计思路：候选补丁、已应用补丁、冲突报告、回滚事件和提案决议都要留下证据链。

### 8.1 用户要求账本

保存用户明确要求、系统推断偏好、确认状态、作用范围和持续检查记录。

### 8.2 角色状态卡

角色状态分为稳定层和动态层。

稳定层包括身份、核心特质、长期目标、道德底线、说话风格、长期秘密和不变量。

动态层包括当前目标、情绪、位置、压力、已知事实、关系边、最近变化和未解决张力。

重要关系变化必须记录原因、章节证据和未解决张力，不能只写标签。

### 8.3 伏笔账本

伏笔应有管理状态和历史动作。

伏笔管理状态包括：

- `Candidate`
- `Active`
- `Recoverable`
- `Recovered`
- `Blocked`
- `Polluted`
- `Transformed`
- `Abandoned`

埋设、强化、复杂化、尝试回收是历史动作，不是管理状态。

核心伏笔不能自动废弃或转化。正文污染核心伏笔时，应阻断章节提交，直到修订或生成重大变更提案。

### 8.4 世界规则账本

世界规则必须区分：

- 规则层级：核心、局部、临时、传闻。
- 真实程度：作者真相、已确认、角色认知、读者暂时认知、错误认知。
- 变更策略：自动、审核、需要重大变更提案。

正文写作者不能随意新增会影响大量后续章节的世界规则。

## 9. 可见性与上下文策略

正文写作者默认按需可见。

系统必须区分：

- 作者或系统知道，但读者暂不应知道。
- 读者已看到，但角色不知道含义。
- 某角色知道，其他角色不知道。
- 已提交正史。
- 候选设定或未确认提案。
- 禁止在本章揭示的信息。

故事规划者、审稿者和故事状态维护者可以在需要时看到更完整的故事正史；正文写作者只能看到当前章节必要信息和硬边界。

上下文重建必须检查：

- 状态版本。
- 索引新鲜度。
- 可见性过滤。
- 泄密风险。
- token 预算。
- 是否包含过期状态。

若上下文重建不可靠，应暂停进入下一章，回到状态同步、上下文过滤、索引重建或滚动复盘。

## 10. 章节事务与提交协议

### 10.1 章节事务原子性

单章事务应具有原子性：要么正文、状态补丁、提交记录和新状态版本一起成为故事正史，要么都暂不落盘到故事正史。

### 10.2 提交门禁

进入 `ChapterCommit` 前必须满足：

- 审稿结论为 `Pass`。
- 如发生局部修订，修订稿已重新检查并通过。
- 没有用户硬要求违反。
- 没有角色不变量破坏。
- 没有核心世界规则冲突。
- 没有核心伏笔污染。
- 状态补丁通过验证并处于 `Pending`，或 `NoStatePatchDeclaration` 已通过检查。
- 没有未处理的高风险变更。
- 没有未处理的确认项；中风险补丁若标记需要确认，必须已经满足确认策略。
- 上下文版本、任务卡版本、正文版本、审稿报告版本和基准状态版本可追溯。

### 10.3 提交步骤

`ChapterCommit` 必须按可恢复事务处理：

每次提交必须有唯一 `commit_intent_id` 或等价幂等键，绑定章节事务 ID、任务卡版本、正文版本、审稿报告版本、补丁或无补丁声明版本、基准状态版本和目标章节号。

同一 `commit_intent_id` 不能绑定不同输入版本。若同一幂等键携带不同任务卡、正文、审稿报告、补丁或基准状态版本，必须视为提交意图冲突，进入暂停处理，不能覆盖既有 `Prepared` 记录。

提交阶段至少包括：

1. `Prepared`：写入提交意图，记录幂等键和所有输入版本，检查基准状态未变化。
2. `Staged`：暂存正文版本、待应用补丁和新状态版本；暂存内容不能作为故事正史对下游可见。
3. `Committed`：写入唯一的 `ChapterCommit` 记录，并把暂存正文版本、新状态版本和提交记录一起提升为故事正史。这个动作是唯一提交点。
4. `Indexed`：基于已提交正史发布或重建派生上下文索引。
5. `Audited`：追加审计事件。

派生索引失败不能让已完成的 `Committed` 回退为未提交，但必须阻止下一章使用过期上下文，直到索引重建成功或生成可用的临时派生视图。

审计事件追加失败不能撤销已经完成的 `Committed`，但必须阻止进入下一章正式事务，直到用同一幂等键补写审计事件或生成审计恢复事件。

如果提交过程中断，系统必须能根据事务记录判断提交是否完成、需要重试、需要回滚，还是需要暂停等待人工确认。

提交重试必须幂等：相同幂等键和相同输入版本只能产生同一个 `ChapterCommit`，不能重复发布正文、重复应用补丁或生成多个状态版本。

### 10.4 失败路由

失败不等于文本没有价值。系统必须按根因路由：

- 任务卡失败：重做任务卡、拆章、压缩目标、补足上下文、回到故事弧复盘或等待用户确认。
- 有创造力的达成方式：如果正文替换场景、动作、对白、局部冲突或章节钩子，但完成章节意图并守住硬边界，应视为正常创作，不进入修订。
- 正文轻微偏离：只修订导致未达成提交条件的问题，尽量保留有效场景、对白、节奏和局部惊喜。
- 正文破坏硬边界：回炉重写、修订或生成重大变更提案。
- 正文出现高价值分岔：生成重大变更提案。
- 审稿未通过：按审稿结论进入修订、重写、提案或暂停。
- 补丁抽取失败：回到补丁提取。
- 补丁证据不足：补证、修订正文或放弃补丁。
- 补丁冲突：生成冲突报告、修订正文或重大变更提案。
- 上下文损坏：从最近一次成功的 `ChapterCommit` 重建。
- 基础设施中断：从最近可判定检查点恢复。
- 工程安全事件：进入 `SecurityPaused`，生成 `SecurityEvent`，阻断相关章节事务，直到恢复条件满足。

### 10.5 回滚与修正

回滚不是普通修订。已提交正史的回退必须记录为显式事件。

系统至少要区分：

- 未提交候选产物作废。
- 最新章节提交撤回。
- 历史事实修正。
- 状态补丁逆转。
- 分支或备选方向归档。

任何回滚或修正都必须留下原因、影响范围、证据和后续需要重建的派生产物。

回滚和修正必须维护依赖关系：

- 最新章节提交撤回后，所有依赖该提交的上下文包、索引、后续候选章节和待应用补丁必须失效。
- 如果已经存在依赖该提交的后续已提交章节，系统必须选择生成新分支、补偿修正事件，或等待人工确认，不能静默删除后续正史。
- 历史事实修正默认是补偿事件或新修正提交，不应无痕重写旧状态版本。
- 状态补丁逆转必须有逆补丁或等价补偿记录，并重新校验影响范围。
- 回滚后必须重建受影响的派生产物，并记录哪些候选产物被作废、保留或转为分支归档。

回滚、撤回和逆补丁操作也必须幂等。相同 `rollback_request_id` 或等价请求键重试时，不能重复追加回滚事件、重复逆转状态、重复作废派生产物或重复归档分支。

## 11. 审稿与修订规则

审稿必须同时检查正确性和章节效果。

正确性检查包括故事正史、任务卡、角色状态、用户要求、世界规则和伏笔状态。

章节效果检查包括读者期待、冲突升级、人物选择、代价、意外信息、期待反转、章节钩子、节奏和可读性。

审稿标准应匹配章节意图。过渡章、余波章、铺垫章、关系沉淀章、反击章和高潮章不应使用完全相同的强冲突模板；章节变化可以是事件推进，也可以是读者理解、人物关系、情绪位置、压力结构或伏笔状态的可追踪变化。

默认修订策略：

- `LocalRevisionRequired` 默认由正文写作者自动修订一次。
- 修订后必须重新检查。
- 连续修订失败应判断根因，而不是无限修订。
- `RewriteRequired` 不应在原文本上局部打补丁，应回到更合适的上游节点。

## 12. 重大变更提案规则

重大变更提案必须包含：

- 触发来源。
- 想保留的新方向。
- 会改变的正史或长期规划。
- 影响范围。
- 可能收益。
- 风险和代价。
- 需要更新的上游产物。
- 被拒绝时的回退方案。

重大变更提案被接受后，系统必须先更新上游产物，再重新检查当前候选章节。不得把原候选正文直接提交为正史。

## 13. 滚动复盘规则

滚动复盘触发条件包括：

- 当前故事弧结束。
- 连续章节节奏同构。
- 伏笔接近最晚回收时机或已经逾期。
- 角色目标失焦。
- 审稿连续多次要求修订或出现退回重写。
- 状态维护者检测到故事正史冲突。
- 用户新要求影响全局方向。
- 上下文长度需要压缩或重建。

连续偏差必须先判断根因：

- 任务卡质量问题。
- 正文执行问题。
- 审稿尺度问题。
- 上下文不足。
- 状态账本冲突。
- 故事弧本身失效。

只有同类根因重复出现，或偏差已经影响故事弧推进时，才上升到滚动复盘。

## 14. 实现施工顺序

实现可以按依赖顺序推进，但每一步都必须服务于完整系统。

建议施工顺序：

1. 文档和账本基底：作品、章节事务、产物版本、审计事件和状态版本。
2. 长线规划产物：需求账本、读者承诺、故事核心引擎、长篇指南针和当前故事弧。
3. 单章事务闭环：上下文重建、章节任务卡、正文稿、审稿报告、修订、状态补丁、补丁校验、提交门禁。
4. `ChapterCommit` 与恢复：原子提交、检查点、失败路由、上下文索引重建。
5. 状态账本完善：角色状态、关系边、伏笔账本、世界规则、用户要求持续检查。
6. 重大变更提案和确认策略。
7. 滚动复盘和长期上下文压缩。
8. 可选增强角色和更细粒度的类型适配。

这些步骤不是产品范围分级，只是施工依赖顺序。

## 15. 验收口径

系统完成后，应能满足以下验收条件：

- 能从简短创作需求生成读者承诺、故事核心引擎、长篇指南针和当前故事弧。
- 能为每章生成非梗概式的章节任务卡，既约束方向又保留正文自由。
- 能生成候选正文、完成自检、审稿、一次修订和重新检查。
- 能从通过审稿的候选正文抽取状态补丁，并基于证据、权限、风险和冲突进行校验。
- 能通过 `ChapterCommit` 原子发布正文、状态补丁、状态版本和提交记录。
- 能在补丁失败、审稿失败、任务卡失败、上下文损坏或基础设施中断时选择正确路由。
- 能区分候选产物、待应用产物、派生产物和故事正史。
- 能保护角色不变量、用户硬要求、世界核心规则和核心伏笔。
- 能处理重大变更提案，不让高风险变化静默入正史。
- 能从最近一次成功的 `ChapterCommit` 重建下一章上下文。
- 能进行滚动复盘，识别连续偏差根因并调整上游产物。
- 能在连续故事弧内保持人物、设定、伏笔和章节推进的可追踪一致性。
- 能生成连续章节，读起来像有节奏、有情绪、有场景生命力的小说，而不是账本驱动的事实列表。
- 能保护工程安全边界，包括密钥不泄露、工具权限不被正文或参考资料越权改变、文件写入范围可审计。
- 能在提交中断、索引失败、回滚和历史修正后保持正史、状态版本、派生产物和候选产物的一致性。

## 16. 待校准参数

以下内容属于参数校准，不影响系统核心边界：

- 滚动复盘的默认章数间隔。
- 不同类型小说的节奏预算默认值。
- 章节吸引力的评分方式。
- 状态补丁风险等级阈值。
- 上下文压缩和召回的默认预算。
- 可选增强角色何时从检查清单拆分为独立角色。

# 写作流程设计理念

## 0. 文档定位

本文是小说写作流程和设计理念的源文档，用来对齐本项目为什么采用这套创作闭环、状态约束和角色分工。

本文不承担最终工程规格职责，也不定义编码智能体的施工规则。当前设计研究目录的产品级文档只保留两份：

- `SPEC.MD`：定义系统要实现什么，包括目标、硬边界、核心闭环、数据产物、状态流转和验收口径。
- `writing-flow-design.md`：解释为什么这样设计，记录写作流程、设计取舍、参考项目吸收点和核心概念。

本文中的 YAML、字段名、流程名和账本结构均为候选结构示例，非最终结构定义。最终结构定义、字段必填性、接口边界和文件布局以后以 `SPEC.MD` 为准。

本项目的目标不是简单调用大模型连续生成章节，而是设计一套可以长期运行的小说创作运行框架：它要让 AI 能持续写长篇，同时尽量避免剧情漂移、人物失真、设定冲突、伏笔遗忘、节奏失控和指令衰减。

## 1. 前置默认决策

以下决策作为当前设计默认前提，不再放在末尾等待裁决。末尾只保留真正需要实验验证的问题。

- 实现策略：完整设计先行，按闭环切片分阶段实现。阶段性交付应是最终系统中的真实闭环片段，而不是临时玩具流程。
- 默认自动化策略：自动模式。普通章节流程全自动推进，只有高风险的重大变更提案阻塞并等待确认。
- 状态模型：采用自定义小说运行账本，吸收 OrisonSpace 的字段白名单、版本、证据、回滚和权限边界思想，但不直接复用其创作字段补丁作为本项目的完整状态模型。
- 正文写作者可见性：默认按需可见。正文写作者只看当前章节必要信息，不直接看到完整幕后真相、终局秘密或未来核心反转。
- 审稿者权限：默认不直接改正文。审稿者输出修订要求和证据，由正文写作者执行一次修订；严重偏离时进入重写或重大变更提案。
- 故事状态维护者：职责上独立，专门负责事实回收、补丁提取和状态校验；运行实现上可以作为章节提交阶段的工具或子流程。
- 高风险变更：不能静默写入故事正史，只能生成重大变更提案。这包括改变主角底线、改变世界核心规则、废弃核心伏笔、提前揭示终局秘密等。

## 2. 核心判断

小说生成不能约束太多，也不能约束太少。

约束太多，正文会变成执行清单，失去临场创造力、人物张力和故事活性。约束太少，长篇会迅速出现设定漂移、角色重置、剧情随意、伏笔污染和用户要求遗忘。

因此，本项目的核心设计原则是：

```text
正文生成阶段要宽，事实落盘阶段要严。
灵感产生阶段要宽，故事正史更新阶段要严。
章节内给正文写作者自由，章节间用状态系统收束。
```

也就是说，系统应该把约束主要放在故事承诺、事实状态、章节任务和审稿回收上，而不是把正文每个场景提前写死。

这个判断决定了后续所有设计：

- 章节任务卡控制方向，不写成正文梗概。
- 正文初稿给正文写作者留足场景、对白、动作、心理和局部反转空间。
- 审稿报告检查章节是否有效，而不把所有创作差异都视为错误。
- 状态变更补丁只回收正文中有证据的事实变化。
- 补丁校验阻止无证据补丁、故事正史冲突和高风险静默变更。
- 章节提交是章节真正进入故事正史的边界。

## 3. 设计来源

参考项目只作为设计来源，不作为本项目的主文档主体，也不直接决定最终工程结构。

### 3.1 oh-story-claudecode

主要吸收读者体验和网文方法论：

- 先明确读者期待，再设计故事。
- 关注类型承诺、爽点、钩子、反转、追读感。
- 每章都要回答“读者为什么还想看下一章”。
- 适合作为创作前的读者承诺分析，以及章节审稿时的吸引力检查。

不直接照搬其提示词材料库形态。本项目需要的是长期运行的创作闭环，不是单次写作技能集合。

### 3.2 character-arc

主要吸收角色连续性思想：

- 角色不是静态设定表，而是持续变化的因果系统。
- 每个主要角色都应有稳定核心和动态状态。
- 章节写作前应知道角色当前目标、情绪、已知信息和关系变化。
- 审稿时要检查角色行为是否符合动机、处境和认知边界。

不把角色系统做成过度审查器。角色弧线应帮助长篇连续，而不是压制场景中的自然反应。

### 3.3 arboris-novel

主要吸收章节任务化思想：

- 每章先有章节使命，而不是直接写正文。
- 单章承担长篇循环中的阶段任务，例如铺垫、压迫、反击、转场、回收。
- 通过节奏预算限制新增设定、人物、伏笔和重大信息数量。
- 正文写作者只看到本章需要知道的上下文，避免误用远期秘密。

不把章节任务细化到替正文写作者决定所有场景。任务卡应提供方向、边界和验收口径，保留正文表现自由。

### 3.4 ainovel-cli

主要吸收长篇自动创作骨架：

- 采用流程协调者 / 故事规划者 / 正文写作者 / 审稿者这类分工。
- 用章节级事务推进长篇：规划、写作、审稿、提交、恢复。
- 保留检查点、上下文预算、章节提交和错误恢复能力。
- 长篇不一次性规划死，而是通过分层大纲和滚动扩纲推进。

本项目在此基础上把写作质量、状态回收、补丁验证和高风险提案作为一等能力。

### 3.5 OrisonSpace

主要吸收本地创作字段、智能体工具边界和故事同步思想：

- 创作字段应结构化保存，如创意简述、世界设定、大纲、角色卡、关系、伏笔等。
- 正文生成后，AI 不应直接任意改故事正史，而应提出最小结构化补丁。
- 状态同步要有安全边界，例如字段白名单、版本、来源、可回退。
- 接口密钥、模型调用、文件写入、智能体权限应分层隔离。

本项目不是完整桌面编辑器，重点先放在长篇自动写作运行框架。故事同步思想需要改造成适合章节事务的小说运行账本。

## 4. 总体流程

本项目采用两层循环：

```text
长线规划循环
  创作需求
  → 读者承诺
  → 故事核心引擎
  → 角色-世界-冲突螺旋校验
  → 长篇指南针
  → 当前故事弧细纲

单章生产循环
  上下文重建
  → 章节任务卡
  → 正文初稿
  → 正文写作者自检
  → 审稿报告
  → 正文写作者一次修订
  → 状态变更补丁
  → 补丁校验
  → 章节提交
  → 下一章上下文重建
```

长线规划负责方向，单章生产负责推进。长线不应把所有章节提前锁死，单章也不应脱离长篇承诺随意发挥。

单章生产循环不应理解为只能向前推进的流水线，而应理解为章节事务。每个节点都有自己的受限出口：有些节点可以进入下一步，有些节点需要局部打磨，有些节点应回到上游重整目标，有些节点只能暂不落盘并形成重大变更提案。路由出口只决定下一步事务动作，不评价文本本身的文学价值。

### 4.1 章节事务

章节事务是本项目最重要的运行单位：

```text
章节任务卡
→ 正文初稿
→ 审稿报告
→ 必要时修订稿
→ 状态变更补丁
→ 补丁校验
→ 章节提交
```

章节提交是章节真正完成的边界。`ChapterCommit` 之前，章节任务卡、正文稿、审稿报告、状态补丁和上下文索引都只是候选产物或待应用产物；只有提交成功后，绑定的正文版本、状态版本、审稿结论和提交记录才一起进入故事正史。

这意味着章节事务应具有原子性：要么正文、状态补丁和提交记录一起成为正史，要么都暂不落盘到正史。若某个环节需要回炉，系统应回到最小相关节点，而不是静默修改已经提交的故事历史。

### 4.2 运行态状态

系统不只需要创作流程，还需要运行态状态机，避免长任务中断或重复生成后无法判断下一步。

候选表达示例，非最终结构定义；最终状态字段以 `SPEC.MD` 为准：

```text
作品阶段：初始化 / 立意 / 大纲 / 写作 / 完成
当前流程：写作 / 审稿 / 修订 / 同步状态 / 调整方向
```

作品阶段表示作品处于哪个大阶段，当前流程表示当前章节事务正在做什么。

### 4.3 核心产物

后续 `SPEC.MD` 可以从本文抽取并标准化以下产物：

- 章节任务卡：定义本章目标、硬边界、戏剧问题、节奏预算和自由区域。
- 正文稿：包括正文初稿和修订稿。
- 审稿报告：给出通过、修订、退回重写、需要重大变更提案等结论和证据。
- 状态变更补丁：从正文抽取已发生事实、角色变化、伏笔变化和世界规则变化；在章节提交前它只是候选补丁或待应用补丁。
- 补丁校验结果：检查证据、权限、冲突和风险等级。
- 重大变更提案：用于处理会改变故事正史的创作选择。
- 章节提交记录：绑定正文版本、任务卡、审稿报告、状态补丁、状态版本和检查点，是章节进入故事正史的发布边界。

## 5. 正文写作者的按需可见性

正文写作者的默认上下文策略是按需可见。

正文写作者应看到当前章节必要信息：

- 当前章节任务卡。
- 相关角色当前状态，包括目标、情绪、已知信息、关系变化和不可破坏的稳定核心。
- 读者承诺。
- 当前故事弧目标。
- 必要世界规则。
- 相关伏笔及其当前文本动作。
- 用户硬要求。
- 禁止揭示、角色禁止知道、硬约束等边界。

正文写作者不直接看到：

- 完整幕后真相。
- 终局秘密。
- 未来核心反转。
- 暂不应进入读者视野的作者真相。
- 与本章无关的远期伏笔解释。

如果需要防泄密，用以下边界表达：

- 禁止揭示：本章不能向读者揭示的内容。
- 角色禁止知道：角色此时不能知道的信息。
- 硬约束：正文不能违反的硬边界。

故事规划者、审稿者、故事状态维护者可以在需要时看到更完整的故事正史，因为它们负责规划、校验和回收；但传给正文写作者的上下文应被过滤。

## 6. 长线规划循环

### 6.1 创作需求

输入可以是一句话，也可以是一段较完整的设定。

系统首先要提取：

- 用户明确要求。
- 用户隐含偏好。
- 类型和读者期待。
- 禁止事项。
- 需要确认的模糊点。

如果用户需求很短，系统可以补全方向，但补全内容应进入可追踪的需求账本，避免后续被当成用户硬要求。

### 6.2 读者承诺

读者承诺是本书对读者的基本许诺。它不是剧情细纲，而是长期校准标准。

候选结构示例，非最终结构定义；最终字段以 `SPEC.MD` 为准：

```yaml
类型: 都市异能 / 规则怪谈 / 玄幻升级 / 文娱逆袭
读者承诺:
  情绪: 压迫后反击
  看点: 主角利用规则漏洞破局
  悬念: 世界异常的来源逐步揭开
  节奏: 快节奏，章节末保留追读钩子
风格边界:
  应该做到:
    - 情绪推进明确
    - 每章有局势变化
  避免:
    - 长篇说明书式设定
    - 主角无代价开挂
```

### 6.3 故事核心引擎

故事核心引擎回答：

- 主角想要什么？
- 主角为什么行动？
- 阻力来自哪里？
- 世界规则如何制造持续冲突？
- 长期悬念是什么？
- 故事为什么可以写成长篇？

一个可运行的长篇故事至少需要：

```text
主角目标 + 长期阻力 + 可重复升级的冲突机制 + 阶段性奖励 + 更大真相
```

如果缺少这些，系统应先补强核心引擎，而不是直接进入章节写作。

### 6.4 角色-世界-冲突螺旋校验

角色、世界和冲突不能分开设计。

校验问题：

- 主角的欲望是否会主动推动剧情？
- 主角的缺陷是否会制造代价和选择？
- 世界规则是否会持续产生问题？
- 反派或阻力是否有自己的行动逻辑？
- 角色变化是否由事件推动，而不是凭空改变？
- 设定是否能产生章节级事件，而不仅是背景说明？

如果某个设定不能制造选择、冲突、代价或悬念，它暂时不应进入核心上下文。

### 6.5 长篇指南针

长篇指南针提供方向感，但不提前写死所有细节。

它应包括：

- 作品类型和读者承诺。
- 主角长期目标。
- 世界核心规则。
- 主要势力或阻力。
- 核心秘密或终局真相。
- 预计大阶段。
- 当前阶段目标。
- 不可破坏的长期约束。

指南针的作用是让后续章节不跑偏，而不是把未来每章固定下来。

### 6.6 当前故事弧细纲

系统只对最近一个故事弧做较细规划。

故事弧可以覆盖 3-10 章，视类型和节奏而定。

故事弧细纲应包括：

- 弧线目标。
- 起始局势。
- 主要冲突升级。
- 关键转折。
- 需要回收或推进的伏笔。
- 角色关系变化。
- 结束时的新局势。

后续故事弧只保留方向和候选可能，不提前锁死。

## 7. 单章生产循环

单章生产循环中的路由逻辑只决定下一步事务动作，不替代正文创作和审稿判断。一个章节暂未达到提交条件，并不等于文本没有价值；它可能只是需要局部打磨、回炉重写、回到规划层重整目标，或把有价值但会改写正史的新方向保存为提案。

### 7.1 章节任务卡

章节任务卡是故事规划者交给正文写作者的核心接口。它控制方向，但保留表现自由。

任务卡应分成三类信息：

- 硬约束：不可违反的边界，如用户硬要求、已发生事实、禁止透露的信息、角色认知边界。
- 目标变化：本章希望造成的变化。它是目标，不应变成逐项机械打勾的剧情清单。
- 创作提示：给正文写作者的创作方向，如戏剧问题、代价、反转空间、应避免的无聊写法。

任务卡本身也需要预检。若任务目标过载、硬约束互相冲突、上下文不足、节奏预算不合理、用户要求不清，或当前故事弧已经不足以支撑这一章，任务卡不应直接交给正文写作者。更合适的去向是重做任务卡、拆章、压缩目标、补足上下文、回到故事弧复盘，或等待用户确认。

候选结构示例，非最终结构定义；最终字段以 `SPEC.MD` 为准：

```yaml
章节编号: 12
阶段: 压迫
大节拍: 规则试探失败，主角发现漏洞可能是陷阱
上一章尾声: 第 11 章结尾，铃声响起前旧教学楼灯光闪烁
本章任务: 让主角第一次意识到规则漏洞也可能是陷阱
戏剧问题: 主角能否利用铃声规律救下队友，还是会反被规则利用？
关键变化: 主角确认管理员可以影响异常触发条件
代价或取舍: 主角为了验证规则，失去队友甲的部分信任
读者期待:
  制造:
    - 读者期待主角找到反制方法
  满足:
    - 回收第 9 章出现的异常铃声
硬约束:
  禁止揭示:
    - 管理员真实身份
    - 世界异常的来源
  角色禁止知道:
    主角:
      - 铃声由管理员主动触发
  用户要求:
    - 主角不能主动牺牲无辜者
目标变化:
  - 主角确认管理员并非普通人
  - 队友甲对主角的信任下降
  - 规则 A 出现例外条件
角色约束:
  主角:
    已知:
      - 铃声与异常出现有关
    未知:
      - 铃声由管理员主动触发
    当前目标:
      - 保护队友并验证规则漏洞
    不能:
      - 主动牺牲无辜者
伏笔:
  埋设:
    - 管理员对旧校徽异常敏感
  强化:
    - 铃声出现前灯光会闪烁
  回收:
    - 第 9 章铃声
  避免:
    - 不要解释旧校徽来源
节奏预算:
  新增有名角色数: 1
  新增重要规则数: 1
  重大揭示数: 1
  伏笔回收数: 1
意外预算:
  允许:
    - 增加 1 个不改变长期故事正史的局部意外
  禁止:
    - 新增会影响后续数十章的核心规则
创作提示:
  场景不应写成:
    - 众人站在原地讨论规则并直接得出结论
  推荐形态:
    - 主角在错误验证中付出代价后，才确认规则漏洞可能是陷阱
自由发挥区:
  - 具体场景调度
  - 对白和心理描写
  - 局部冲突设计
  - 章节末钩子
验收标准:
  - 本章至少产生一种可感知变化，可以是冲突升级、人物选择、代价、意外信息、期待反转、读者理解推进、人物关系变化或情绪位置变化
  - 本章结束后，读者对局势、人物、悬念或情绪压力的理解不能完全回到开章前状态
预期状态变化:
  - 主角已知事实增加“铃声与灯光闪烁有关”
  - 队友甲对主角的信任下降，并记录原因
  - “异常前铃声”伏笔进入已回收或尝试回收状态
```

章节任务卡不是正文梗概。它只规定本章要造成什么变化、哪些边界不能破、这一章应该产生什么戏剧张力。

正文写作者可以改变达成方式，例如换场景、换动作、换对话、换局部冲突，但应保留硬边界、主要目标变化和章节验收条件。

### 7.2 正文创作

正文写作者的职责是把任务卡变成可读正文。

正文写作者应拥有充分自由：

- 场景选择。
- 对白设计。
- 行动细节。
- 心理描写。
- 局部冲突。
- 微反转。
- 章节钩子。
- 段落节奏。

正文写作者不应擅自改变：

- 主角核心人设。
- 世界硬规则。
- 已发生事实。
- 用户硬要求。
- 长期主线方向。
- 核心伏笔状态。
- 重要角色命运。

正文进入审稿前，正文写作者应先做轻量自检。自检只处理明显遗漏、硬约束误触、局部连续性、格式和风格问题，不替代审稿者，也不负责结构性审稿。自检的目的不是缩小创作自由，而是避免把显而易见的小问题交给后续节点放大。

正文完成后，系统应区分三类变化：

- 有创造力的达成方式：替换场景、动作、对白、局部冲突或章节钩子，只要完成章节目标并守住硬边界，就应被视为正常创作。
- 破坏硬边界的偏离：违反用户硬要求、已发生事实、角色认知边界、世界硬规则或核心伏笔状态时，应进入局部修订、回炉重写，或回到任务卡重新规划。
- 值得改变故事方向的分岔：如果正文提出了更强的故事方向，但它会改变故事正史或长期规划，应保存为重大变更提案，而不是静默写入正文。

重大变更提案被批准后，也不应直接把原候选正文纳入正史。系统应先更新长篇指南针、当前故事弧、任务卡或状态账本，再基于新的上游重新检查、修订或重写章节。

### 7.3 审稿与修订

审稿者不应只检查错别字，而要检查故事是否有效。

审稿维度：

- 是否完成章节任务。
- 是否违反硬事实。
- 角色行为是否符合当前动机、信息和关系。
- 读者期待是否被制造、满足或延迟。
- 本章是否有冲突升级、人物选择、代价、意外信息或期待反转。
- 本章是否出现可追踪变化，而不是局势完全回到原点。
- 节奏是否拖沓。
- 新增设定是否超出节奏预算。
- 伏笔是否被错误回收或污染。
- 是否有明显的指令遗漏。
- 章节末是否有继续阅读的理由。

审稿者应把检查分成两类：

```text
一致性检查：是否正确，是否遵守故事正史、任务卡、角色状态和用户要求。
可读性检查：是否好看，是否有冲突、选择、代价、意外和阅读驱动力。
```

如果章节只是合规推进，但没有任何可感知的变化，不能直接通过，应进入修订。这里的变化不只包括冲突升级、人物选择、代价、意外信息或期待反转，也可以是读者理解、人物关系、情绪位置、压力结构或伏笔状态的可追踪变化。

审稿结果候选枚举如下，最终以 `SPEC.MD` 为准：

```text
通过：可以提交。
修订：需要局部修订；默认最多自动修订一次。
需要重大变更提案：正文提出了重大变更，需要进入确认流程。
退回重写：严重偏离，需要重写。
```

默认流程中，审稿者不直接改正文。它输出修订要求、问题证据和目标效果，由正文写作者执行一次修订，再重新检查。

审稿结论应对应明确的后续动作：

- 通过：进入状态同步。
- 修订：进入最小有效修订，只处理导致未达成提交条件的问题，尽量保留原稿中有效的场景、对白、节奏和局部惊喜。
- 退回重写：用于章节目标整体失效、硬边界破坏严重、角色或设定明显漂移，或局部修订无法救回的情况。
- 需要重大变更提案：用于正文中出现有价值但会改写故事正史的分岔。
- 暂停提交：用于需求矛盾、用户硬要求不清、重大提案待确认或上游信息不足的情况。

### 7.4 状态同步

章节通过审稿后，故事状态维护者从已通过审稿的候选正文中抽取结构化变化。

状态同步不是重新创作，而是事实回收。

状态同步分为四步，但在章节提交前，补丁只能处于候选或待应用状态：

```text
提取补丁：从已通过审稿的候选正文中抽取候选状态补丁。
校验补丁：检查补丁是否有正文证据，是否违反故事正史，是否超出权限。
准备应用补丁：将通过验证的补丁作为待应用补丁，等待 ChapterCommit 原子合入状态库。
准备上下文索引：根据待应用补丁生成候选索引材料，等待提交后发布或重建。
```

需要更新：

- 已发生事件。
- 角色当前位置、目标、情绪、已知信息。
- 关系变化。
- 世界规则新增或修正。
- 伏笔状态变化。
- 用户要求满足情况。
- 下一章应继承的未完成事项。

候选结构示例，非最终结构定义；最终字段以 `SPEC.MD` 为准：

```yaml
补丁编号: 第12章状态同步
来源章节: 12
基准状态版本: 状态版本31
字段版本: 1
生成者: 故事状态维护者
风险等级: 低
置信度: 0.86
变更:
  角色:
    主角:
      动态状态.当前目标: 验证管理员触发铃声的方式
      动态状态.已知事实:
        新增:
          - 铃声出现前灯光会闪烁
  伏笔:
    异常前铃声:
      状态: 已回收
      证据: 第 12 章确认铃声与异常出现有关
  世界:
    规则:
      新增:
        - 规则 A 在灯光闪烁后存在例外条件
证据:
  - 章节: 12
    引文或摘要: 铃声响起前灯光连续闪烁，随后异常发生
是否需要确认: 否
回滚提示: 从状态版本32撤销第12章状态同步
```

状态补丁需要可审计、可回退，不能让模型随意重写整个设定库。

待应用补丁规则：

- 低风险补丁可以进入待应用补丁，例如角色当前地点、当前目标、普通关系变化、已发生事件摘要。
- 中风险补丁可以进入待应用补丁但应标记风险，例如普通伏笔推进、局部规则例外、重要关系转折。
- 高风险补丁转重大变更提案，例如改变主角底线、改变世界核心规则、废弃核心伏笔、提前揭示终局秘密。
- 没有正文证据的补丁不能进入待应用补丁。
- 与现有故事正史冲突的补丁不能静默覆盖旧状态，只能生成冲突报告或重大变更提案。
- 章节提交前应通过状态补丁验证，否则不能进入章节提交。

补丁未通过不自动否定正文文本价值，但会阻断章节提交。若问题来自抽取格式、证据定位或字段权限，应回到补丁提取与校验；若问题来自正文确实制造了故事正史冲突、核心伏笔污染或高风险不可逆变化，则应进入正文修订、回炉重写或重大变更提案。

### 7.5 章节提交门禁

章节提交为故事正史前，应同时满足：

- 审稿结论为通过；如果曾经需要修订，必须完成修订并重新审稿为通过。
- 修订、退回重写、需要重大变更提案或暂停提交都不能直接进入章节提交。
- 没有用户硬要求违反。
- 没有角色不变量破坏。
- 没有核心世界规则冲突。
- 没有核心伏笔污染。
- 状态补丁通过验证并处于待应用状态，或已经按 `SPEC.MD` 的无补丁声明规则确认本章无需补丁。
- 如触发高风险变更，进入重大变更提案，而不是直接提交。

`ChapterCommit` 应把正文版本、审稿结论、待应用状态补丁、状态版本和检查点作为同一事务发布。若门禁未达成提交条件，内容问题回到修订或重写，状态问题回到补丁提取、补丁校验或冲突报告，高风险问题进入重大变更提案，基础设施中断或上下文缺失则从最近可判定检查点恢复。

### 7.6 下一章上下文重建

下一章写作前，流程协调者根据已提交的故事正史重建上下文。上下文重建是派生过程，不是新的故事正史；如果索引损坏、状态缺失或上下文超预算，应从最近一次成功的 `ChapterCommit` 重建，而不是猜测当前故事状态。

上下文应包含：

- 本书读者承诺。
- 当前故事弧目标。
- 最近章节摘要。
- 当前章节任务卡。
- 相关角色当前状态。
- 相关世界规则。
- 相关伏笔。
- 用户硬要求。

不相关的远期秘密和幕后设定不应全部塞给正文写作者，以免泄密、误用或污染当前章节。

上下文重建还应检查状态版本、索引新鲜度、可见性过滤和泄密风险。若重建结果会把角色不应知道的信息、读者不应看到的真相或过期状态交给正文写作者，应暂停进入下一章，回到状态同步、上下文过滤或滚动复盘处理。

## 8. 角色分工

### 8.1 流程协调者

职责：

- 判断当前流程阶段。
- 调度故事规划者、正文写作者、审稿者、故事状态维护者。
- 处理局部打磨、回炉重写、提案确认、暂停提交和断点恢复。
- 控制上下文预算。
- 决定何时进入滚动复盘。

流程协调者不应直接写正文，除非是在极简运行模式下合并角色。

### 8.2 故事规划者

职责：

- 设计读者承诺。
- 提炼故事核心引擎。
- 维护长篇指南针。
- 规划当前故事弧。
- 生成章节任务卡。
- 控制节奏预算。

故事规划者负责方向和结构，但不替正文写作者决定所有表达细节。

### 8.3 正文写作者

职责：

- 根据章节任务卡写正文。
- 在自由区域内创造场景、对白、冲突和细节。
- 保持角色声音和叙事风格。
- 制造章节内的情绪流动和阅读钩子。

正文写作者是创造力最强的角色，但要遵守硬事实和任务边界。

### 8.4 审稿者

职责：

- 检查章节任务完成度。
- 检查可读性、节奏、爽点、悬念。
- 检查角色一致性。
- 检查设定和伏笔问题。
- 提出局部修订要求。

审稿者不应把所有创作差异都视为错误。它要区分“有意的张力”和“真实的漂移”。

### 8.5 故事状态维护者

职责：

- 基于已通过审稿的候选正文、章节任务卡、旧状态和审稿结论，抽取候选状态补丁。
- 校验候选补丁是否有正文证据。
- 检查用户要求、角色不变量、世界规则和伏笔状态是否被违反。
- 输出候选状态补丁、不变量检查报告和冲突列表。
- 维护事实状态、角色动态状态、伏笔账本、世界规则和用户要求账本。

禁止事项：

- 不写正文。
- 不设计未来剧情。
- 不判断章节是否好看。
- 不删除故事正史。
- 不在无证据时修改账本。
- 不静默覆盖冲突状态。

故事状态维护者不负责写好看，只负责让长篇可持续。它在职责上独立，工程上可以作为提交阶段的工具、子流程或智能体。

### 8.6 可选增强角色

以下角色先作为审稿者或故事状态维护者的检查清单，后续再按复杂度拆分：

- 角色维护者：专门检查角色弧线、关系、口吻和动机。
- 伏笔维护者：专门管理伏笔生命周期和回收时机。
- 读者模拟器：专门模拟读者期待、困惑、爽感和追读欲。

避免一开始形成过重的委员会式写作流程。

## 9. 约束分层

系统采用四层约束。

### 9.1 故事正史层

硬约束，不可随意改变。

包括：

- 用户硬要求。
- 类型底线，例如作品不能从规则怪谈静默变成轻喜剧日常。
- 主角核心身份、欲望、底线和能力边界。
- 世界核心规则。
- 已发生事实。
- 角色已知信息边界。
- 重要角色不可逆状态。
- 核心伏笔状态。
- 核心秘密揭示权限。
- 禁止事项。

违反故事正史的正文应修订或进入重大变更提案。

### 9.2 创作意图层

软约束，用于指导方向。

包括：

- 卷目标。
- 章节目标。
- 情绪曲线。
- 爽点方向。
- 文风偏好。
- 角色弧线倾向。
- 角色关系趋势。
- 普通伏笔推进窗口。
- 类型手法，例如偏悬疑、偏反转、偏升级、偏群像。
- 节奏建议。

创作意图可以被创造性改写，但不能无理由消失。

### 9.3 正文创作层

自由创作层。

包括：

- 场景细节。
- 对白。
- 动作。
- 心理。
- 局部冲突。
- 小道具。
- 临时配角。
- 微反转。
- 叙事节奏。

这是正文写作者发挥的主要空间。

### 9.4 重大变更提案层

需要确认的重大变更。

包括：

- 改变主角长期目标。
- 改变主角核心性格或道德底线。
- 改变世界核心规则。
- 重要角色死亡、黑化、背叛或洗白。
- 提前揭示核心秘密。
- 提前回收或废弃核心伏笔。
- 大幅偏离故事弧目标。
- 改变作品类型或文风方向。
- 改变叙事视角结构。
- 引入长期主角级新角色。
- 永久改变能力系统上限。
- 把临时配角升级为主线角色。
- 让角色做出违反底线但剧情想保留的行为。
- 把本应延后的真相变成读者明确信息。

重大变更提案可以由正文写作者、审稿者或故事规划者提出，但不能静默落盘。在默认自动模式下，普通流程全自动，重大变更提案是主要阻塞点。

重大变更提案被接受后，应先回写到相应的上游产物，例如长篇指南针、当前故事弧、章节任务卡或状态账本，再重新检查候选章节是否仍然成立。提案本身不是故事正史，不能绕过章节提交边界。

## 10. 状态账本

状态账本是自定义小说运行账本，不是 OrisonSpace 创作字段补丁的直接复刻。它应吸收字段白名单、版本、证据、回滚和权限边界思想，但围绕章节事务和长篇连续性重建。

状态账本应采用追加式审计思路：候选补丁、已应用补丁、冲突报告、回滚事件和提案决议都应留下证据链。重试同一节点时，应生成新的候选版本，而不是静默覆盖旧尝试；上游产物被重做后，下游候选产物应重新确认或作废；已提交正史的回退应作为显式修正或回滚事件记录，而不是普通修订流程中的无痕改写。

以下示例用于说明设计方向，不代表最终结构定义。

### 10.1 用户要求账本

用户要求账本用于避免用户指令在长篇中衰减。

候选结构示例，非最终结构定义；最终字段以 `SPEC.MD` 为准：

```yaml
要求编号: 要求001
来源: 用户
类型: 硬要求
性质: 持续不变量
强度: 必须遵守
优先级: 100
作用范围: 全书
内容: 主角不能主动牺牲无辜者
状态: 生效中
原始文本: 主角不能主动牺牲无辜者
上次检查章节: 12
检查结果: 通过
证据:
  - 初始需求
```

其中“持续不变量”不是一次性任务，不能简单标记为“已满足”。例如“主角不能主动牺牲无辜者”应在每章审稿和状态同步时持续检查。

### 10.2 角色状态卡

角色状态分为稳定层和动态层。

候选结构示例，非最终结构定义；最终字段以 `SPEC.MD` 为准：

```yaml
角色编号: 主角
稳定层:
  身份: 异常调查新人
  核心特质:
    - 冷静
    - 不愿牺牲无辜者
  长期目标: 找出异常世界的来源
  道德底线: 不主动牺牲无辜者
  说话风格: 简短、克制、带试探
  口吻锚点:
    - 回答先确认事实，再给判断
  不变量:
    - 不主动牺牲无辜者
  秘密:
    - 童年曾接触过异常事件
动态层:
  当前目标: 验证管理员是否能触发铃声
  当前情绪: 警惕
  当前位置: 旧教学楼
  当前压力: 队友甲开始怀疑主角判断
  弧线阶段: 第一次主动试探规则
  动机桥接: 保护队友，同时验证管理员是否能干预规则
  已知事实:
    - 铃声与异常有关
  关系边:
    - 对象: 队友甲
      信任: 下降
      冲突: 怀疑主角利用队友试探规则
      原因: 第 12 章验证规则时让队友甲陷入危险
      最近变化: 第12章
      证据: 队友甲质问主角是否早知道风险
      未解决张力: 队友甲暂时配合，但不再完全信任
  最近变化:
    - 第 12 章发现规则 A 存在例外条件
  最近出场章节: 12
```

正文写作者需要动态层来写当前章节，故事规划者需要稳定层来规划角色弧线。

角色关系不宜只写成“信任下降”这类标签。每次重要关系变化都要记录原因、章节证据和未解决张力，否则长篇后续很容易关系重置。

### 10.3 伏笔账本

伏笔应有管理状态和文本动作。

候选状态集合，非最终结构定义；最终枚举以 `SPEC.MD` 为准：

```text
候选 / 生效中 / 可回收 / 已回收 / 被阻塞 / 被污染 / 已转化 / 已废弃
```

“埋设”“强化”“复杂化”“尝试回收”不是管理状态，而是写入历史记录的文本动作。

候选结构示例，非最终结构定义；最终字段以 `SPEC.MD` 为准：

```yaml
伏笔编号: 异常前铃声
状态: 生效中
重要性: 核心
埋设章节: 第9章
所属故事弧: 旧教学楼
相关对象:
  - 管理员
  - 旧教学楼
可见性:
  读者可见: 是
  读者知道含义: 否
预期回收: 铃声是管理员触发异常的信号
回收条件:
  - 主角至少两次观察到铃声与异常的关联
  - 管理员干预规则的可能性已被暗示
最晚回收章节: 第15章
阻塞依赖:
  - 管理员真实身份不能提前揭示
废弃需要: 重大变更提案
历史:
  - 章节: 9
    动作: 埋设
    证据: 异常前出现铃声
  - 章节: 11
    动作: 强化
    证据: 铃声前灯光闪烁
```

伏笔不是越多越好。系统应控制伏笔数量，避免每个细节都被登记成未来负担。

核心伏笔不能自动废弃或转化。如果正文污染伏笔意图，应进入污染状态并阻断章节提交，直到修订或生成重大变更提案。

### 10.4 世界规则账本

世界规则要区分硬规则、角色误解和读者暂时认知。

候选结构示例，非最终结构定义；最终字段以 `SPEC.MD` 为准：

```yaml
规则编号: 规则A
规则层级: 核心
作用范围: 旧教学楼
真实程度: 已确认
变更策略: 需要重大变更提案
优先级: 80
读者可见: 是
内容: 夜间进入旧教学楼后，听见铃声的人会被异常标记
例外:
  - 灯光闪烁后规则存在例外条件
冲突规则: []
证据:
  - 章节: 4
    摘要: 第一次进入旧教学楼时，铃声后异常出现
引入章节: 第4章
更新章节:
  - 第12章
```

正文写作者不应随意新增会影响大量后续章节的世界规则。

世界规则要区分：

- 规则层级：核心 / 局部 / 临时 / 传闻
- 真实程度：作者真相 / 已确认 / 角色认知 / 读者暂时认知 / 错误认知
- 变更策略：自动 / 审核 / 需要重大变更提案

## 11. 滚动复盘

系统应按节奏和复杂度周期性进入滚动复盘。具体章数间隔属于 `SPEC.MD` 中的待校准参数，不在本文固定。

触发条件：

- 当前故事弧结束。
- 连续 N 章节奏同构，冲突模式重复。
- 伏笔接近最晚回收章节或已经逾期。
- 角色目标失焦，章节推进和主角欲望脱节。
- 审稿者连续多次要求修订或出现退回重写。
- 故事状态维护者检测到故事正史冲突。
- 用户新要求影响全局方向。
- 上下文长度需要压缩或重建。

连续偏差不应被简单理解为单章失败，而应先判断根因：任务卡质量问题、正文执行问题、审稿尺度问题、上下文不足、状态账本冲突，还是故事弧本身失效。只有同类根因重复出现，或偏差已经影响故事弧推进时，才上升到滚动复盘。

复盘内容：

- 当前故事弧是否仍然有效。
- 读者承诺是否保持。
- 主角目标是否清晰。
- 主要角色关系是否有推进。
- 伏笔是否过多或遗忘。
- 章节节奏是否重复。
- 是否需要扩展下一故事弧。
- 是否需要提出重大变更提案。

滚动复盘不应每章都做成完整世界重构，否则成本高且干扰创作。

滚动复盘的输出也应先作为候选调整、下一故事弧规划或重大变更提案。除非经过后续章节事务提交，它不应自动改写已提交的故事正史。

## 12. 首个完整闭环切片

首个完整闭环切片不追求一次性完成全部能力，但应使用最终系统中的真实产物、真实状态流转和真实提交边界。

建议范围：

```text
单书
单线主角
单故事弧滚动
章节级事务闭环
连续产出 5-10 章并保持状态一致
```

优先纳入：

- 读者承诺。
- 长篇指南针。
- 当前故事弧细纲。
- 章节任务卡。
- 正文初稿。
- 审稿报告。
- 正文写作者一次修订。
- 故事状态维护者状态同步。
- 状态变更补丁。
- 补丁校验。
- 章节提交。
- 用户要求账本。
- 角色状态卡。
- 伏笔账本。
- 世界规则账本。
- 下一章上下文重建。

切片内可简化：

- 角色状态卡只保留稳定层、动态层、关系边和少量不变量。
- 伏笔账本只登记明确承诺未来回收的伏笔。
- 世界规则账本只登记硬规则、局部规则和已确认例外。
- 修订默认最多自动执行一次。
- 默认使用一套通用章节任务卡候选结构，类型差异放进“读者期待”“创作提示”和“节奏预算”。

暂缓：

- 完整桌面编辑器。
- 复杂多人实时协作。
- 过度细化的角色心理模型。
- 每章多轮委员会式审稿。
- 自动拆分大量专业子智能体。
- 过早做全书每章细纲。
- 多主角多线并行。
- 为不同类型小说实现多套完全独立任务卡模板。

首个完整闭环切片的目标是先跑通稳定的长篇循环，而不是一次性实现所有理想功能。

默认自动化策略保留三档作为后续配置方向：

```text
自动模式：默认全自动，仅重大变更提案阻塞。
故事弧复盘模式：每个故事弧复盘时允许人工确认。
严格模式：高风险状态补丁和重大变更提案都需要人工确认。
```

当前默认目标是自动模式，但内部要保留重大变更提案、风险等级和回滚记录，用于处理重大不可逆变化。

## 13. 后续实验问题

以下问题仍需通过样章、连续章节生成和状态回放验证：

- 如何量化“章节好看”。
- 如何更准确地区分角色自然变化和角色漂移。
- 上下文成本和质量之间的默认平衡点。
- 滚动复盘的默认频率如何随类型和复杂度变化。
- 何时把角色维护者、伏笔维护者、读者模拟器从检查清单拆成独立角色。
- 候选结构收敛到 `SPEC.MD` 时，哪些字段应保留，哪些只作为提示词材料。

## 14. 暂定结论

本项目的写作流程应以 `ainovel-cli` 的长篇运行框架为骨架，以 `arboris-novel` 的章节任务化为单章接口，以 `character-arc` 的角色状态管理保证人物连续性，以 `oh-story-claudecode` 的读者体验方法检查章节吸引力，以 `OrisonSpace` 的故事同步思想控制正文后的状态回写。

最终目标不是让智能体机械执行大纲，而是形成一种可持续的创作循环：

```text
方向明确，但不锁死表达。
状态严格，但不压死创作。
章节有任务，但正文有生命。
长篇能自动推进，也能在关键节点收束和修正。
```

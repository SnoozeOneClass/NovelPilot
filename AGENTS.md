# Codex 项目协作说明

## 项目事实来源

本项目是一个面向长篇小说自动生成的 AI Agent Harness。

在规划架构、编写代码、修改 schema、创建提示词或拆分任务之前，必须先阅读根目录下的两份文档：

- `SPEC.MD`：项目权威规格。它定义目标、硬边界、核心闭环、数据产物、状态流转、提交协议、失败路由和验收口径。
- `writing-flow-design.md`：设计理念说明。它解释为什么采用这套写作流程、状态约束和角色分工。

如果两份文档存在差异，以 `SPEC.MD` 作为实现依据。

## 核心产品边界

必须保留这些不变量：

- 正文生成阶段要给创作自由；事实和状态落盘阶段必须严格。
- 候选产物不是故事正史。
- 只有成功的 `ChapterCommit` 才能把绑定的正文版本、状态版本、审稿结论和提交记录提升为故事正史。
- 状态变更必须有证据，证据只能来自用户输入、已提交正文、提案决议、人工确认或显式系统事件。
- 不能静默改写已经提交的故事正史。必须使用显式修正、回滚、逆补丁、分支归档或新的章节事务。
- 高风险变化必须变成 `MajorChangeProposal` 或等待确认，不能静默进入正史。
- 上下文包、索引等派生产物必须能从已提交正史中重建。
- 工程安全和故事可见性必须分开处理。API key、模型凭据、本地敏感配置和工具权限不能进入正文、上下文包、审稿报告、状态补丁或提交记录。

## 实现策略

按依赖顺序施工，使用最终系统中的真实闭环切片，不做后续需要推翻的临时玩具流程。

第一个有价值的 MVP 切片应聚焦：

- 单书。
- 单线主角。
- 单个活跃故事弧。
- 章节级事务闭环。
- 连续生成 5-10 章，并保持状态一致。

优先实现：

- 作品/项目模型。
- 章节事务模型。
- 产物版本。
- 审计事件。
- 状态版本。
- `RequirementLedger`、`ReaderPromise`、`StoryEngine`、`NovelCompass` 和 `ArcPlan`。
- `ChapterTaskCard`、正文稿、自检、审稿报告、状态补丁、补丁校验、无补丁声明和 `ChapterCommit`。
- 从最近一次成功提交中重建下一章上下文。

暂缓实现：

- 完整桌面编辑器。
- 多人实时协作。
- 多主角、多线并行。
- 大量专业子智能体。
- 多套完全独立的类型模板。

## 多 Agent 分析与 Review

为了优先保证正确性，本项目接受较重的分析和 Review 流程。凡是进行代码修改，默认都必须启用双重保险；不要因为改动看起来小就自动跳过，除非用户在当前任务中明确豁免。

代码修改前，默认开启多个 subagent 做并行分析，再由主 agent 汇总判断后动手。分析重点包括：

- 当前方案是否符合 `SPEC.MD` 和 `writing-flow-design.md`。
- 是否破坏故事正史、候选产物、派生产物和 `ChapterCommit` 的边界。
- 是否有更小、更稳、更贴合当前里程碑的实现路径。
- 是否存在状态机、幂等性、证据链、失败恢复或工程安全隐患。

代码修改完成后，先由主 agent 做常规代码审查和必要测试，再开启多个 subagent 做设计 Review，最后由主 agent 汇总处理。常规代码审查重点关注真实 diff、bug、测试缺口、类型错误、接口破坏和运行风险；subagent 设计 Review 重点包括：

- 修改是否符合原始设计和当前里程碑。
- 是否引入隐藏耦合、过早抽象、临时流程或后续需要推翻的实现。
- 是否遗漏测试、失败路由、边界条件或安全检查。
- 是否存在候选产物泄漏进正史、状态无证据更新、提交非原子等风险。

如果 Review 发现问题，应优先修复；无法立即修复时，必须明确记录原因、风险和后续处理建议。

纯文档修改、格式化和注释修改可以不启动多 subagent，但必须在回复中说明跳过原因。涉及运行行为、构建配置、测试配置、schema、状态机、提交协议、模型调用或安全边界的改动，即使文件很少，也按代码修改处理。

## 工程规则

- 修改范围要贴合当前里程碑，不做无关重构。
- 除非用户明确要求，不要改写 `SPEC.MD` 或 `writing-flow-design.md`。
- 文件保持 UTF-8 编码。
- 核心产物优先使用结构化 schema、状态机和解析器，不要依赖临时字符串拼接。
- 测试应覆盖状态流转、提交幂等性、证据要求和失败路由。
- 不要因为某个技术栈熟悉就直接选用；必须结合规格和当前里程碑说明理由。

## Git 自动化规则

Goal 模式下，Git 用于工程回溯和阶段审计；Git commit 不等于产品验收，也不等于小说系统里的 `ChapterCommit`。

- 依赖 Codex 自身权限系统处理高风险 Git 操作，本文件只定义项目级使用策略。
- 正式 Goal 开始前，必须先有可回退的干净基线；没有初始提交时，先提交项目规格基线。
- 每个 Goal 使用独立分支，例如 `goal/v0.1-closed-loop`，不要直接在 `main` 上跑长任务。
- Goal 执行中允许自动创建本地 checkpoint commit，用于保存阶段成果、测试修复、Review 修复和中断现场。
- checkpoint commit 必须写清 Goal、阶段、验证状态和设计依据。
- 如果工作区混有用户未确认改动、敏感文件、无关文件或冲突状态，不要自动提交，应先停下来说明。
- `push`、合并到 `main`、改写历史、删除分支/tag 等发布或破坏性操作，必须由用户在当前上下文中明确要求，并遵守 Codex 权限确认。

## Review 重点

审查代码或方案时，优先关注：

- 是否破坏故事正史边界。
- 状态变更是否缺少证据。
- `ChapterCommit` 是否失去原子性或幂等性。
- 候选产物是否泄漏进已提交正史。
- 密钥、外部文件、网络调用或工具权限是否处理不安全。
- 审稿失败、补丁校验失败、索引失败、提交中断和回滚是否缺少恢复路径。
